# Workflow Spec Recorder 产品体验评审

评审角色：产品经理  
评审日期：2026-06-11  
评审范围：README、CONTEXT、TODO、AGENTS、操作手册、桌面端主要界面、核心导出与泛化生成逻辑。

## 1. 真实产品定位

Workflow Spec Recorder 不是单纯的录屏工具，也不是 RPA 执行器。它的真实定位是：面向自动化工作流生成工具的需求采集与规格生成工具。

用户通过它记录少量“代表性人工操作”，再补充粗略需求、输入样例、输出样例、业务资料和截图证据，最终生成一份可交付给 APA / AI-RPA / 自动化工作流生成工具理解的需求文档。当前内置的是 APA 需求文档模板，但 APA 是第一种交付模板，不应被理解为产品边界。

产品价值不在于把用户点击复刻成固定脚本，而在于把示例点击、页面结构、文件资料和业务目标综合成可泛化的自动化需求：包括动态集合遍历、输入输出结构、异常处理、等待策略、日志和验收标准。

一句话定位建议：

> Workflow Spec Recorder 帮助用户把人工业务操作示例和业务资料转换成自动化工具可理解、可执行、可复核的需求规格。

## 2. 当前完整使用流程

### 2.1 安装

1. 用户在 Windows 上运行安装包，例如 `WorkflowSpecRecorder-Setup-0.2.5.exe`。
2. 安装目录默认是 `%LOCALAPPDATA%\Programs\WorkflowSpecRecorder`，安装向导允许用户选择其他有写入权限的位置。
3. 安装完成后可从开始菜单或桌面快捷方式启动 `Workflow Spec Recorder`。
4. 安装包会携带 Chrome 扩展目录和使用说明，但浏览器扩展不能被普通安装包静默安装。

### 2.2 加载 Chrome 扩展

1. 用户启动桌面端。
2. 点击桌面端中的 `打开插件目录`，或从开始菜单打开 `打开 Chrome 扩展目录`。
3. 在 Chrome 地址栏打开 `chrome://extensions/`。
4. 开启右上角“开发者模式”。
5. 点击“加载已解压的扩展程序”。
6. 选择安装目录下的 `extension` 文件夹。
7. 加载成功后，扩展名为 `Workflow Spec Recorder桥接扩展`。
8. 用户可以打开扩展弹窗查看状态：桌面端未连接、等待开始录制、录制中、暂停等。

### 2.3 检测连接

1. 桌面端启动本地采集服务 `127.0.0.1:18765`。
2. 用户点击 `检测插件` 或手册中的 `检测插件连接`。
3. 如果最近收到扩展心跳，桌面端显示“浏览器插件在线”，并显示最近心跳时间。
4. 如果没有收到心跳，桌面端会打开本地检测页 `http://127.0.0.1:18765/extension-check`。
5. 本地检测页每秒轮询 `/health`，只有 `extensionRecentlySeen=true` 才表示扩展真的连上。
6. 需要注意：本地服务可用不等于扩展在线。扩展最近心跳窗口约为 20 秒，用户需要打开扩展弹窗、刷新业务页面或重新加载扩展来触发心跳。

### 2.4 录制

1. 在 `录制 / 编辑` 页填写：
   - 流程名称
   - 业务目标
   - 前置条件
2. 点击顶部 `开始/继续录制`。
3. 到 Chrome 中按正常人工方式操作业务系统。
4. Chrome 扩展采集页面打开、跳转、点击、输入、下拉选择、文件上传和下载完成/失败事件。
5. 桌面采集层会尝试记录 Windows 文件保存/打开对话框、客户端窗口、Excel/ERP 等非浏览器页面中的点击、输入和确认键。
6. 桌面端显示：
   - 当前录制状态
   - 已记录步骤数
   - 收到浏览器事件数
   - 忽略事件数
   - 最近一次事件是否被记录
7. 扩展图标显示：
   - `REC`：桌面端在线并正在录制
   - `等`：桌面端在线但未开始录制
   - `停`：桌面端在线，录制已暂停
   - `OFF`：扩展未连上桌面端
8. 用户可点击 `暂停` 检查步骤，也可点击 `停止` 结束录制。

### 2.5 编辑步骤

1. 用户在步骤时间线中选择步骤。
2. 中间区域展示步骤标题、成功判定和截图预览。
3. 右侧步骤属性区可编辑或查看：
   - 业务标题
   - 动作类型
   - 页面标题
   - 页面地址
   - 目标元素名称
   - CSS 选择器
   - 变量名
   - 固定值
   - 敏感字段
   - 截图路径
   - 成功判定
   - 备注
4. 对生成质量最关键的是业务标题、变量名和成功判定。它们帮助后续生成器理解“业务意图”，而不是只复述点击。

### 2.6 导出原始录制包

1. 用户点击顶部 `导出录制包`。手册中称为 `导出原始录制包`。
2. 系统默认导出到 `%USERPROFILE%\Documents\ApaFlowRecorder\Exports\`。
3. 每次导出生成一个独立文件夹，包含：
   - `workflow.json`
   - `APA需求文档.md`
   - `assets/` 截图资源
4. 导出后，系统会自动把 `workflow.json`、`APA需求文档.md` 和截图资源加入 `泛化需求` 页资料集。
5. 用户可以点击 `打开导出目录` 查看原始包；该按钮只有完成一次原始包导出后才可用。

### 2.7 添加资料

1. 用户切换到 `泛化需求` 页。
2. 点击 `添加资料`，可一次选择多个需求文档、输入样例、输出样例、截图说明或参考附件。
3. 已导出的原始录制包会自动出现在资料路径中，不需要用户手工再次选择。
4. 支持读取的主要资料类型：
   - Excel：读取 Sheet、行列数、表头和预览行。
   - Word：读取正文，并提取内嵌图片作为视觉证据。
   - PPT：读取幻灯片文字，并提取内嵌图片。
   - TXT、Markdown、CSV、JSON、XML、HTML、LOG：读取文本预览。
   - PNG、JPG、JPEG、WEBP：作为视觉资料发送给支持图像输入的模型。
   - PDF：当前主要记录文件名和大小，不读取正文。
5. 用户可在 `补充要求` 中说明业务重点、输出要求和异常处理偏好。

### 2.8 配置模型

1. 在 `泛化需求` 页的 `大模型设置` 区域，用户可选择：
   - `MiniMax 国内预设`
   - `DeepSeek 预设`
2. 用户可手动填写：
   - 服务商
   - Base URL
   - 模型名
   - API Key
3. API Key 会明文保存到当前 Windows 用户配置文件。
4. 当前界面没有单独的“测试模型连接”按钮。CONTEXT 中已经定义了 `Connection Test` 术语，但当前产品路径中尚未落成明显入口。
5. API Key 为空时仍可生成文档，但会走规则兜底模式，质量和业务贴合度通常低于 LLM 生成。
6. 用户可编辑提示词模板，并执行：
   - 保存模板
   - 恢复默认
   - 打开模板文件
7. 模板中的 `{{recorded_example}}`、`{{source_materials}}`、`{{extra_instruction}}` 是关键占位符，应保留。

### 2.9 生成 APA 需求文档

1. 用户确认资料、补充要求和模型配置后，点击 `生成 APA 需求文档`。
2. 系统读取资料，构建提示词，收集可用截图和资料图片。
3. 如果模型配置完整，调用 OpenAI-compatible Chat Completions 接口生成 Markdown。
4. 如果模型输出缺少核心章节，或包含被禁止的分析过程章节，系统会请求模型按模板修正。
5. 如果模型未配置，系统走规则兜底生成。
6. 当前最终文档被收敛到固定结构：
   - `# 项目需求描述`
   - `## 项目目标`
   - `## 流程步骤`
   - `## 流程输入`
   - `## 流程输出`
   - `## 约束与异常处理`

### 2.10 打开导出目录

1. 生成完成后，系统默认导出到 `%USERPROFILE%\Documents\ApaFlowRecorder\GeneralizedExports\`。
2. 每次生成会创建独立文件夹，包含：
   - `APA-generalized-requirements.md`
   - `workflow-spec.json`
   - `llm-prompt.txt`
   - `generation-mode.txt`
3. 用户点击 `打开泛化导出目录` 查看结果。
4. 当前建议路径是：人工阅读 `APA-generalized-requirements.md`，必要时修改后复制到 APA 工作区，让 APA 基于该文档生成流程。

## 3. 首次使用最容易卡住的地方

1. 产品命名和入口命名不完全统一。用户会同时看到 `Workflow Spec Recorder`、`APA流程录制器`、`WorkflowSpecRecorder`、`插件`、`扩展`、`泛化需求` 等词，第一次使用时很难判断这些是不是同一个东西。
2. Chrome 扩展加载是最大激活阻塞。用户需要理解安装目录、打开插件目录、Chrome 开发者模式、加载已解压扩展、刷新扩展、刷新业务页面、打开扩展弹窗等多个非日常动作。
3. `本地采集服务已启动` 容易被误解为 `浏览器插件已连接`。实际产品里这两个状态不同，但第一次使用者通常不会主动区分。
4. `检测插件` 这个按钮语义偏短，手册中又写 `检测插件连接`，用户可能不知道检测失败后应该操作 Chrome、扩展卡片还是桌面端。
5. 用户可能点击了 `开始/继续录制`，但没有真实扩展心跳或业务页面未刷新，导致有操作却没有新增步骤。
6. 用户可能在暂停状态或未开始录制状态操作浏览器，事件会进入忽略计数，但他们未必知道“忽略”意味着什么。
7. 步骤编辑区专业字段较多。变量名、固定值、成功判定、敏感字段、选择器对目标用户是有价值的，但第一次使用时缺少“哪些字段必须改、哪些可不管”的优先级提示。
8. `导出录制包` 和 `生成 APA 需求文档` 是两个不同输出。用户可能以为导出原始包就是最终需求，或不知道导出后还要去 `泛化需求` 页生成最终 APA 文档。
9. `泛化需求` 页的主按钮在左侧滚动区域底部。资料、模型、提示词模板都在同一个滚动面板里，第一次用户可能需要滚到底才发现生成按钮。
10. 模型配置没有显性的连接测试入口。用户只有在点击生成后才知道 Base URL、模型名或 API Key 是否可用。
11. API Key 明文保存是必要安全提示，但当前没有把“这是生成文档用的 LLM Key，不是扩展 Key，也不是 APA 运行 Key”讲到足够靠近输入框的位置。
12. PDF 已加入但不读正文这一限制容易造成质量预期落差。用户看到“添加资料成功”后，可能以为 PDF 内容已经参与分析。
13. 图片数量限制没有面向用户说明。核心逻辑最多给模型附带 12 张图，长录制或多资料图片时，用户可能不知道后续图片只是以文字/文件信息辅助。
14. 生成完成后的下一步还不够产品化。用户需要知道应该打开哪份 md、如何检查、如何交给 APA，而不是只看到导出目录。

## 4. 问题分型

### 4.1 产品路径问题

1. 首次激活路径过长：安装桌面端和加载扩展被拆在两个环境里完成，中间没有一个强引导式检查清单。
2. 核心流程被拆成两个页签，但“录制代表性示例”和“生成最终需求”的关系仍需要用户自己理解。
3. 原始录制包和泛化需求文档的价值层级没有被充分区分。原始包是中间材料，泛化需求文档才是面向 APA 的主要交付物。
4. 无界面资料处理流程是重要能力，但入口被放在 `泛化需求` 页说明文案里，用户可能以为必须先录制。
5. 从“生成完成”到“在 APA 工作区使用文档”的最后一公里仍停留在手册说明，没有成为应用内路径。
6. 模型配置、资料添加、提示词模板和生成动作都在同一页，信息优先级不够清晰，新用户会把高级配置当成必填项。

### 4.2 交互设计问题

1. `检测插件`、`打开插件目录`、`导出录制包` 等按钮语言偏工程化，不能完整表达动作结果。
2. 扩展连接失败后虽然会打开检测页，但桌面端没有在同一位置给出下一步动作闭环，例如“打开插件目录 -> 打开 chrome://extensions -> 刷新扩展 -> 再检测”。
3. `打开导出目录` 和 `打开泛化导出目录` 只有在对应导出后启用；禁用时缺少原因提示。
4. 录制状态、扩展状态、本地服务状态、事件计数同时出现，状态之间的主次关系不明确。
5. 生成按钮位于长滚动面板底部，用户完成资料添加和模型配置后可能需要寻找主操作。
6. 步骤编辑没有“建议补全程度”。用户不知道仅修改业务标题是否足够，还是必须填写变量名、成功判定和备注。
7. 添加资料后路径文本框可编辑，但缺少每个文件的读取状态列表。用户不容易发现某个文件不存在、PDF 未读取正文或 Office 图片已提取。
8. 模型调用失败只显示异常文本，对非技术用户缺少可执行恢复动作。

### 4.3 视觉表现问题

1. 当前视觉基调清爽，信息密度适合桌面工具，但首页级主次略平均，核心下一步没有足够强的视觉聚焦。
2. 顶部工具栏承载了录制、检测、导出等高频动作，但 `打开插件目录` 在页内状态卡中，首次扩展配置链路被视觉上拆散。
3. `泛化需求` 左侧区域信息过长，资料、模型、补充要求、提示词模板、生成按钮连续堆叠，首屏难以判断哪里是必填，哪里是高级项。
4. 状态卡使用了颜色区分录制状态，但扩展连接状态、模型配置状态和生成状态没有形成统一的状态体系。
5. 右侧生成说明目前更像文档摘录，生成前后视觉变化不强，无法很好承接“我现在该做什么”。

### 4.4 文案说明问题

1. `插件` 和 `扩展` 混用。Chrome 语境里用户更熟悉“扩展”，建议统一为“Chrome 扩展”，必要时首次出现写“Chrome 扩展（插件）”。
2. `泛化需求` 是准确但偏专家的词。第一次使用者更容易理解“生成最终需求文档”或“生成 APA 需求”。
3. `导出录制包` 容易被误解为最终交付。应说明“导出原始录制包，作为生成最终需求的资料”。
4. `API Key（明文保存到当前 Windows 用户配置文件）` 是安全提示，但缺少“只用于调用大模型生成需求文档”的边界说明。
5. `不填 API Key 时会使用规则兜底` 需要更明确地说明质量预期：可生成结构化草稿，但复杂业务建议配置模型。
6. PDF 限制、图片数量限制和纯文本模型限制需要在添加资料或生成前更靠近用户决策点。
7. `忽略事件数` 对用户不透明。它背后可能是未开始、暂停、扩展状态异常或事件类型过滤，当前文案不足以帮助用户自救。
8. 手册中有少量错别字和口语化表达，例如“滑倒最下面”，会影响正式交付感。

### 4.5 技术限制问题

1. Chrome 未上架本地扩展不能由普通安装包静默安装，首次必须手动加载。
2. 扩展连接依赖本地 `127.0.0.1:18765` 服务和扩展心跳；健康接口可用不代表扩展连接成功。
3. 扩展心跳的“最近在线”窗口较短，用户一段时间没有触发心跳后可能看到离线。
4. 桌面采集依赖 Windows UI Automation 和窗口信息，复杂自绘客户端可能只能记录窗口级信息。
5. PDF 当前不读取正文，只记录文件名和大小。
6. 发给模型的图片数量有限，当前最多附带 12 张可用图片。
7. LLM 调用依赖用户提供 OpenAI-compatible Base URL、模型和 API Key，网络、鉴权、模型多模态能力都会影响生成质量。
8. 当前不会自动执行 RPA，也不会处理目标站点的登录态、验证码、反爬和限流执行问题。
9. API Key 以明文保存到当前 Windows 用户配置文件，需要用户理解本地安全边界。

## 5. 第一阶段最小改造范围

第一阶段目标不是扩大能力，而是降低首次成功路径的摩擦：让新用户能在一次会话中完成“扩展在线 -> 录到步骤 -> 导出原始包 -> 生成最终需求 -> 找到结果并知道下一步”。

### 5.1 必做范围

1. 统一关键命名。
   - 产品名统一为 `Workflow Spec Recorder`。
   - 用户可见的浏览器组件统一称为 `Chrome 扩展`。
   - `导出录制包` 明确为 `导出原始录制包`。
   - `泛化需求` 在关键入口旁补充解释为“生成最终 APA 需求文档”。

2. 增加首次使用检查清单。
   - 检查项包括：本地服务已启动、Chrome 扩展已加载、最近收到扩展心跳、已开始录制、已记录步骤、已导出原始包、已生成最终需求。
   - 每个未完成项提供一个直接动作或一句明确下一步。

3. 强化 Chrome 扩展连接失败闭环。
   - 在桌面端连接状态区域直接说明失败原因和下一步。
   - 把 `打开插件目录`、`检测插件连接` 和 `chrome://extensions` 操作路径放在同一组引导里。
   - 明确“本地服务在线不等于扩展在线”。

4. 明确原始包与最终需求的关系。
   - 导出完成后提示：原始包已加入资料集，下一步是切到生成页生成 `APA-generalized-requirements.md`。
   - 在生成页顶部显示“录制包已加入/未加入”的状态。

5. 降低生成页首用复杂度。
   - 把资料、模型、生成按钮分成更清晰的必填路径和高级设置。
   - 提示词模板默认折叠或标记为高级配置。
   - 生成按钮保持容易发现，不依赖用户滚动到很深的位置。

6. 增加模型配置前的预期管理。
   - 靠近 API Key 说明：这是 LLM 生成需求文档用，不是 Chrome 扩展凭证，也不是 APA 运行凭证。
   - 说明不填 API Key 会走规则兜底，适合快速草稿，不适合复杂页面和复杂资料。
   - 增加或预留“测试模型连接”入口，避免用户到生成阶段才发现配置不可用。

7. 改善资料读取状态反馈。
   - 添加资料后显示文件清单、类型、读取结果和限制。
   - PDF 显示“未读取正文，建议转 Word/TXT”。
   - 图片显示“会作为视觉证据参与，数量可能受限”。

8. 生成完成后的下一步提示。
   - 明确打开目录后应查看 `APA-generalized-requirements.md`。
   - 提醒用户先人工检查，再交给 APA。
   - 同时保留 `workflow-spec.json`、`llm-prompt.txt`、`generation-mode.txt` 作为排查材料。

### 5.2 暂不纳入第一阶段

1. 不扩大采集范围，不处理复杂自绘客户端 OCR。
2. 不新增 RPA 执行能力。
3. 不做多模板市场化能力，只保留 APA 作为当前内置模板。
4. 不把提示词模板编辑做成完整模板管理系统。
5. 不做复杂账号、团队协作或云端同步。
6. 不把安装包变成自动安装 Chrome 扩展，因为这受 Chrome 安全机制限制。

### 5.3 第一阶段验收标准

1. 新用户按应用内引导能完成 Chrome 扩展加载，并明确看到“Chrome 扩展在线”。
2. 用户开始录制后，至少能在普通网页上录到点击或输入步骤，并知道如果没有步骤应检查什么。
3. 用户能理解 `导出原始录制包` 是中间资料，不是最终交付。
4. 用户能在 `泛化需求` 页明确看到资料状态、模型状态和主生成动作。
5. 生成成功后，用户能直接找到 `APA-generalized-requirements.md` 并理解它是交给 APA 的主文档。
6. 对 PDF、图片数量、纯文本模型、API Key 明文保存等限制，用户在操作前已经看到清晰提示。
